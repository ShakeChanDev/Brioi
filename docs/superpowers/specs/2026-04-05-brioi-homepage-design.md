# Brioi API Homepage Design Spec

**Date:** 2026-04-05  
**Project:** Brioi API 首页  
**Scope:** 纯 HTML / CSS / JS 单页落地页  
**Primary Goal:** 用带有强排版个性的首页，清晰介绍 Brioi API，并把用户直接导向可售套餐购买页。  

## 1. Product Context

Brioi API 不是纯 API 点数站，卖的是 GPT / Codex 等官方客户端的订阅通道与镜像接入。首页必须在短时间内让用户理解三件事：

1. 这是官方客户端可直接使用的接入方案，不是缩水版。
2. 套餐价格清晰，适合直接购买，不需要先读复杂教程。
3. Brioi API 的核心竞争点是价格划算，其次才是“官方可用”和“购买后可直接上手”。

## 2. Design References And Translation

### 2.1 Visual Reference

首页视觉语言参考 [typos.twsp.app](https://typos.twsp.app/)，但不是复制其紫色方案或原始内容。要吸收的是以下结构特征：

- 白底、黑字、高反差强调色
- 夸张的大字号标题和极强的版式留白
- 轻微玩味感和对话式内容组织
- 少量、有目的的视觉戏剧性，而不是传统 SaaS 功能宫格

### 2.2 Pricing Structure Reference

价格区结构参考 [univibe.cc/pricing](https://www.univibe.cc/pricing) 的逻辑，而不是样式本身。要保留的是：

- 分类切换清楚
- 同一分类内的套餐卡尺寸统一
- 主推方案明确

## 3. Brand Direction

### 3.1 Brand Name

首页品牌名固定为 `Brioi API`。

### 3.2 Tone

整体语气定义为：

- 直给
- 有一点挑衅感
- 有设计感，但不装
- 先谈价格，再补可信度

不使用“企业官网式”的保守文案，也不做过度夸张的营销喊话。文案要短、硬、好扫读。

## 4. Confirmed Design Decisions

### 4.1 Homepage Priority

首页定位为“品牌感与转化并重，但整体仍以成交为目标”。

### 4.2 Primary Selling Point

用户首先记住的卖点固定为：`价格划算，套餐清晰`。

### 4.3 Conversion Path

首页所有核心 CTA 均导向单独购买页，不在首页中嵌入注册、登录或教程流程。  
统一购买链接路径定义为 `/buy`。

### 4.4 Content Scope

首页只保留四类内容：

- 网站介绍
- 订阅价格
- FAQ
- 页尾品牌信息

不加入接入教程、平台支持列表区块、博客入口或复杂功能对比区块。

### 4.5 Frontend Stack

实现方式固定为纯 `HTML / CSS / JS`，不使用 React、Next.js 或构建框架。

## 5. Visual System

### 5.1 Base Palette

整体色彩方案如下：

- 页面主背景：`#FFFFFF`
- 主文字色：`#000000`
- 次级文字色：`#4B5563`
- 分隔和浅底卡片：`#F3F4F6`
- 主强调色：`#5BE139`
- 强调色亮阶：`#8DFC4F`
- 深色面板：`#000000`

### 5.2 Accent Usage Rules

绿色只出现在这些位置：

- Hero 标题中的重点词
- 对话气泡中的主动回应块
- 主推套餐价格数字
- tab 激活状态的小面积强调
- 少量渐变文字或标签

不能把整个页面涂成绿色，也不能把所有按钮都做成绿色。整体仍然以白底黑字为主，绿色负责“击中视线”。

### 5.3 Typography

排版风格以“超大标题 + 简洁正文 + 少量等宽标签”为核心：

- 标题：使用一款紧凑、现代的无衬线字体，字重 800 左右，字距偏紧
- 正文：使用清晰的无衬线字体，保证中英文混排可读性
- 辅助标签：使用等宽字体，模拟编辑标记或产品标签

实现阶段应采用具有更强个性的字体组合，而不是默认系统字体堆栈。

### 5.4 Shape Language

- 主卡片圆角偏大，接近 `20px` 到 `30px`
- 深色卡与浅灰卡混用，但都保持统一圆角体系
- 不使用锐利边框和复杂拟物阴影
- 阴影只用在主推价格卡上，普通区块保持轻量

## 6. Information Architecture

页面按以下顺序组织：

1. 顶部导航
2. Hero 首屏
3. 网站介绍区
4. 价格区
5. FAQ 区
6. 页尾

整页为单页滚动结构，不做多屏切换，不做路由级分段。

## 7. Section Design

### 7.1 Top Navigation

顶部导航保持极简：

- 左侧：`Brioi API`
- 右侧：单个主按钮 `立即购买`

导航不放复杂菜单。点击按钮直接跳转 `/buy`。

导航视觉要求：

- 白底
- 黑字
- 不要厚重阴影
- 顶部留足呼吸空间

### 7.2 Hero Section

Hero 是整页最重要的视觉锚点，结构固定为“左侧大标题与对话内容，右侧单一主推价格卡”。

#### Hero Title

主标题固定为三行结构：

```text
不想再为
官方客户端
付更贵？
```

其中“付更贵？”使用强调色绿色。

#### Hero Support Copy

Hero 不使用传统副标题大段文案，改为 3 个对话式内容块：

1. `我想直接用 GPT / Codex。`
2. `那就别买贵的。一个订阅，直接上。`
3. `是官方客户端镜像接入，不是缩水版，也不是只卖 API 点数。`

排列方式：

- 第 1 条和第 3 条为浅灰气泡
- 第 2 条为绿色强调气泡
- 整体类似对话消息流，但比聊天界面更克制

#### Hero Primary Card

右侧只保留一个主推卡片：

- 套餐：`Plus 月卡`
- 价格：`¥99`
- 额度：`100 刀 / 天`
- 时长：`30 天有效`
- 按钮：`购买 Plus 月卡`

该卡片是首屏唯一价格展示核心，不在其下方追加第二组小价格块。

#### Hero Bottom Statement

Hero 下方增加一句居中总结：

`一个订阅，直接覆盖 GPT / Codex`

其中产品名可使用轻量渐变或绿色强调，但面积必须小于主标题。

### 7.3 Website Introduction Section

网站介绍区不做 3 到 4 个通用功能卡，而是做成编辑感内容块。该区的目标是用最少内容补足用户的理解与信任。

固定表达的 3 个信息点：

1. `官方客户端直接可用`
2. `不是缩水版，也不是纯点数站`
3. `购买动作放在独立购买页，接入细节不在首页展开`

布局要求：

- 使用大标题 + 2 到 3 个短块说明
- 允许穿插少量标签样式和小号等宽文字
- 不做宫格图标区

### 7.4 Pricing Section

价格区采用“分类 tab + 分类内统一套餐卡”的结构。

#### Pricing Heading

价格区标题固定为：

```text
选一个方案。
直接开用。
```

第二行中的“直接开用。”使用绿色强调。

#### Pricing Tabs

价格区顶部使用 2 个 tab：

- `体验套餐`
- `月卡套餐`

默认激活 `月卡套餐`，因为首页的首屏主推就是 `Plus 月卡`，默认状态需要与首屏一致。

#### Tab 1: 体验套餐

展示 2 张统一卡片：

1. `日卡`
   - 价格：`免费`
   - 说明：`20 刀 免费使用`
   - CTA：`购买日卡`

2. `周卡`
   - 价格：`¥29.9`
   - 说明：`100 刀 / 天`
   - CTA：`购买周卡`

在该分类中，周卡可以作为该 tab 的次级推荐项，用深底或绿色标签强调。

#### Tab 2: 月卡套餐

展示 3 张统一卡片：

1. `Plus 月卡`
   - 价格：`¥99`
   - 说明：`100 刀 / 天`
   - 时长：`30 天`
   - 推荐标签：`推荐`
   - CTA：`购买 Plus`

2. `Pro 月卡`
   - 价格：`¥199`
   - 说明：`200 刀 / 天`
   - 时长：`30 天`
   - CTA：`购买 Pro`

3. `Max 月卡`
   - 价格：`¥499`
   - 说明：`600 刀 / 天`
   - 时长：`30 天`
   - CTA：`购买 Max`

#### Pricing Card Rules

- 同一 tab 下的卡片高度统一
- 同一 tab 下的卡片结构统一
- 只有主推方案使用深底 + 绿色价格
- 卡片上不放多余角标和复杂对比文案
- 所有购买按钮直接跳转 `/buy`

### 7.5 FAQ Section

FAQ 区保留大号问答块，不做小字号折叠列表作为默认形态。

区标题固定为：

```text
你会问的，
先回答。
```

第二行中的“先回答。”使用绿色强调。

FAQ 条目数量控制在 4 到 6 条，首版固定为以下 5 条：

1. `支持哪些客户端？`
   - 回答：目前只支持 GPT、Codex，其他暂不支持。

2. `这是官方客户端接入，还是 API 点数？`
   - 回答：这是官方客户端镜像接入与订阅通道，不是只卖 API 点数。

3. `额度怎么计算？`
   - 回答：周卡和月卡均按每日额度展示，月卡有效期为 30 天。

4. `购买后去哪里？`
   - 回答：所有首页 CTA 统一进入购买页 `/buy`。

5. `哪个套餐最适合大多数人？`
   - 回答：默认推荐 `Plus 月卡 ¥99`，适合大多数长期使用场景。

### 7.6 Footer

页尾保持极简，只保留：

- `Brioi API`
- 简短一句品牌描述
- 一个购买入口

不做大而全页脚，不放无关链接集合。

## 8. Interaction Design

### 8.1 Required Interactions

仅实现以下交互：

1. 价格区 tab 切换
2. 页面内锚点滚动
3. 所有购买按钮跳转 `/buy`

### 8.2 Optional Interactions

允许加入少量轻动效，但必须克制：

- Hero 文案入场淡入上移
- tab 切换时价格卡轻微淡入
- hover 只做轻微位移或阴影变化

禁止加入复杂鼠标跟随、夸张视差或大面积滚动特效。

## 9. Responsive Behavior

### 9.1 Desktop

- Hero 采用左右分栏
- 左侧标题与对话内容占视觉主导
- 右侧价格卡保持固定权重，不压过标题

### 9.2 Mobile

- Hero 改为单列堆叠
- 顺序为：标题 -> 对话块 -> 主推价格卡
- 价格区 tab 仍保留，但卡片改为单列排列
- FAQ 保持单列大问答块，不改成手风琴作为默认形态

### 9.3 Spacing Rules

移动端不因为屏幕小就压缩成“普通模板站”。  
标题仍然要足够大，只是调整换行和节奏。  
留白优先级高于一次性塞入更多元素。

## 10. File Structure

实现时文件结构固定为：

```text
/Users/Shake/Documents/App/Brioi/index.html
/Users/Shake/Documents/App/Brioi/styles.css
/Users/Shake/Documents/App/Brioi/main.js
```

购买页作为首页 CTA 的目标页，应创建：

```text
/Users/Shake/Documents/App/Brioi/buy.html
```

## 11. Progressive Enhancement And Failure States

即使 JS 失效，页面也必须保持可用。

具体要求：

- 默认展示 `月卡套餐` tab 内容
- tab 按钮在无 JS 场景下退化为普通按钮样式，不影响主内容阅读
- 所有购买按钮仍为标准链接
- FAQ 因为默认是展开的大问答块，所以不依赖 JS

## 12. Testing Requirements

实现完成后至少验证以下内容：

1. 桌面端与移动端布局是否符合设计层级
2. Hero 标题和主卡是否在首屏内形成明确主次关系
3. 价格区 tab 切换是否正确显示对应套餐
4. 所有 CTA 是否统一跳转 `/buy`
5. FAQ 文案是否完整且没有折断布局
6. 页面在无 JS 情况下是否仍然可读可点击

## 13. Non-Goals

本次首页不包含以下内容：

- 接入教程页内容
- 登录或注册流程
- 多平台功能细节对比
- 后台控制台功能展示
- 博客、案例、客户 logo 区

## 14. Final Design Summary

Brioi API 首页应表现为一个带有强排版性格的成交型落地页：

- 视觉上继承 `typos.twsp.app` 的高反差与排版张力
- 价格结构上借鉴 `univibe` 的分类与主推逻辑
- 内容上只保留介绍、价格和 FAQ
- 转化上统一导向 `/buy`

最终目标不是做一个“看起来像 SaaS 模板”的首页，而是做一个用户扫一眼就知道卖什么、多少钱、该点哪里的首页。
